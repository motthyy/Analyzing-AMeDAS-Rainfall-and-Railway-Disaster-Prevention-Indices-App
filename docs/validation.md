# 検証記録

本書は、開発時に実際に行った検証内容と結果を記録する。「検証済み」と記載した項目のみが
実行時に確認されたものであり、記載のない挙動は未検証として扱うこと。

検証日: 2026年7月18日。環境: Windows 11、Python 3.11.9、`.venv`仮想環境。

## 1. 単体テスト

```powershell
.\.venv\Scripts\python.exe -m pytest -q
```

結果: **45 passed**（0 failed）。対象は以下の通り（`tests/`配下）。

- `test_continuous_rainfall.py`: 12時間無降雨リセット連続雨量・0.3mm閾値・欠測明け再初期化
- `test_rolling_rainfall.py`: 24時間移動雨量の単純合計一致・先頭NaN・欠測を含む窓のNaN化
- `test_effective_rainfall.py`: 実効雨量の初期値・指数減衰・半減期での半減・欠測明け再初期化
- `test_soil_tank.py`: 10分雨量の均等分配・タンク貯留量非負・無降雨中の総貯留量非増加・
  水収支・手計算ケースとの一致・YAML係数読込
- `test_annual_maxima.py`: 3種類の年区切り・年境界の正しい振り分け・最大値日時保存・完全率計算
- `test_gumbel.py`: SciPy MLEとの一致・積率法の数式一致・T=1年のNaN・確率年の単調増加・
  ブートストラップの再現性・退化リサンプル（実データで発生したOverflowError）への耐性
- `test_csv_parser.py`: CP932デコード・複数行ヘッダー・24時正規化・欠測・品質情報・
  数値変換・複数ファイル間の重複検出

## 2. 実データによるエンドツーエンド検証

以下の手順を、`.venv`内のPythonから気象庁本番サーバーへ実際にHTTPリクエストを送信して確認した
（PowerShellの`Invoke-WebRequest`と、本アプリの`JmaDirectClient`の両方で検証）。

### 2.1 セッション確立・地点一覧取得

- `GET https://www.data.jma.go.jp/gmd/risk/obsdl/index.php` → 200、`ci_session`等のCookie取得を確認。
- `POST .../top/station`（`pd=44`、東京都）→ 地点一覧HTML取得、UTF-8での正しいデコードを確認。
  小河内(`a0365`)・八王子(`a0366`)・調布(`a0367`、廃止地点、観測終了日の抽出)等を確認。

### 2.2 CSVダウンロード・パース

- station=`a0365`（小河内）、2024年1月1〜2日分を`ymdLiteral=0`で取得し、
  年・月・日・時が分割された列形式であること、24時が literal な "24" として
  得られることを確認。
- 1970年（観測開始前）を指定すると、品質コード「0」・値欄が空になることを確認
  （開始日時探索アルゴリズムの前提として利用）。

### 2.3 パイプライン全体（ダウンロード→統合→正規化→指標計算→年最大値→ガンベル→ブートストラップ）

station=`a0365`、2021〜2023年の3年分（各年8760行）を実際にダウンロードし、以下を確認した。

```
1) ダウンロード: 各年8760行取得
2) 統合(merge_hourly_frames): 26,280行、欠測11時間、値競合0件
3) 正規化(reindex + 閾値処理): 26,280行
4) 指標計算:
   最大連続雨量(12h reset)   = 187.5 mm
   最大24時間移動雨量        = 166.5 mm
   最大実効雨量(半減期6h)    = 72.54 mm
   最大推定土壌雨量指数      = 120.37
5) 年最大値(暦年、実効雨量6h):
   2021年: 40.23mm (2021-07-02 12:00)
   2022年: 49.05mm (2022-09-24 02:00)
   2023年: 72.54mm (2023-06-02 16:00)
   2024年: 0.60mm  (2024-01-01 00:00、12/31 24時が翌日0時へ繰り込まれた1時間のみのため)
6) ガンベル分布(最尤法): mu=27.13, beta=25.78、100年確率雨量=145.7mm
   ブートストラップ(200回、シード42): 100年値の95%信頼区間 [58.4, 175.1]
```

この結果、途中で発生した `OverflowError`（ブートストラップのリサンプルが退化した
標本になった場合にSciPyのフィッティングが数値的に破綻する不具合）を発見し、
`statistics/bootstrap.py` の例外捕捉を修正した（`ValueError`のみ捕捉していたのを
`OverflowError`/`RuntimeError`/`FloatingPointError`/`ZeroDivisionError`も捕捉するよう変更）。
この不具合は回帰テスト `test_bootstrap_handles_degenerate_resamples_without_crashing` として
`tests/test_gumbel.py` に追加した。

### 2.4 CP932/UTF-8エンコーディング

CSVダウンロードのレスポンスはCP932であること、地点一覧HTMLのレスポンスはUTF-8であることを
それぞれ確認した（同じサイトのエンドポイントでも文字コードが異なる点に注意）。

### 2.5 ダウンロードジョブ管理（SQLite）・冪等性

`DownloadManager` + `JobRepository`（一時DB・一時保存先）を用いて、station=`a0365`、
2024-01-01〜2024-03-31（部分年、1ジョブ）で以下を確認した。

```
1) plan_jobs() でジョブ1件を作成
2) run() で実際にCSVを取得し、SUCCESS状態・保存ファイルパス・生CSVファイルの実在を確認
3) 同一期間で再度 plan_jobs() を呼んでも新規ジョブが重複作成されないこと（UNIQUE制約）
4) 再取得後、get_actionable_jobs() が0件（既にSUCCESSのため再取得対象にならない）こと
```

いずれも想定通りの結果となった。ただし、意図的に通信を失敗させての自動分割
（6か月→3か月→1か月→7日）・指数バックオフ再試行の実行時確認は行っていない
（4節に記載の通り未検証）。

## 4. 未検証事項（重要・正直な申告）

以下は、コードレビュー・設計上の整合性確認は行ったが、**実行時の網羅的な検証は行っていない**。

- 地点マスタの全都道府県一括取得（約47都道府県分のループ処理）。東京都1県のみ実地点で確認。
- 開始日時探索アルゴリズム（`start_date_finder.py`）の、実際に長期欠測がある地点・
  非単調な観測パターンを持つ地点での動作。
- `jma/playwright_client.py`（Playwright予備手段）の実行時動作。direct_clientと同一の
  リクエスト仕様を実装しているが、Playwright経由での実行はコードレビューのみで、
  実際にブラウザを起動しての動作確認は行っていない。

- 数十年規模（例: 1974年〜現在の50年分）の長期データでの実行時間・メモリ使用量、
  および全項目Excelブック出力の実行時間。

  （3年分のダウンロードはpipeline関数を直接呼び出して確認したが、UIボタン経由での
  長時間バッチ処理は未検証）。
- Windows以外のOS、Python 3.11以外のバージョンでの動作。
- 気象官署種別（`s`プレフィックス地点）での「現象なし情報」列を含むCSVの実データ解析
